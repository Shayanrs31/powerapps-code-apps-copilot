# Team Directory — Power Apps Code App

A people-directory SPA that searches your organisation via the **Office 365 Users** connector, connected through a **connection reference** for ALM-safe deployment across Dev / Test / Prod.

---

## Architecture

```
TeamDirectory (component)
  └── useTeamDirectory (hook)
        └── Office365UsersService  ← auto-generated by PAC CLI
              └── Office 365 Users connector
                    └── Connection Reference  ← solution component / ALM portable
```

---

## Prerequisites

- Power Platform environment with Code Apps enabled
- Node.js LTS
- PAC CLI (`%LOCALAPPDATA%\Microsoft\PowerAppsCLI\pac.cmd`)
- A **solution** in Power Apps with a **Connection Reference** pointing to an Office 365 Users connection

---

## Step 1 — Initialise the code app

```bash
cd my-team-directory
npx power-apps init --displayName "Team Directory" --environmentId <your-env-id>
```

Sign in with your Power Platform account when prompted.

---

## Step 2 — Authenticate PAC CLI

```bash
pac auth create --environment <your-env-url>
# e.g. pac auth create --environment https://myorg.crm.dynamics.com
```

---

## Step 3 — Find your solution ID and connection reference name

**List solutions:**

```bash
pac solution list
```

Copy the `Id` of the solution that contains your Office 365 Users connection reference.

**List connection references in that solution:**

```bash
pac code list-connection-references -env <your-env-url> -s <solutionId>
```

Copy the **Logical Name** of the connection reference (e.g. `new_office365users_ref`).

---

## Step 4 — Add the data source via connection reference

```bash
pac code add-data-source \
  -a "shared_office365users" \
  -cr <connectionReferenceLogicalName> \
  -s <solutionId>
```

This generates:
- `src/generated/services/Office365UsersService.ts`
- `src/generated/models/Office365UsersModel.ts`

> Never edit these files manually. If the connector schema changes, delete and re-add.

---

## Step 5 — Run locally

```bash
npm run dev
```

Open the **Local Play** URL in the same browser profile as your Power Platform tenant.
Grant "local network access" if Chrome/Edge prompts you (required since Dec 2025).

## Known issues & resolutions

### `pac code add-data-source -cr` → "Failed to resolve connection ID"

**Symptoms**

```
Error: Failed to resolve connection ID for reference '<logicalName>' in solution '<solutionId>'
```

Occurs with PAC CLI v1.52.1 through v2.4.1 when using `pac code add-data-source -a <api> -cr <logicalName> -s <solutionId>`.

**Root cause**

PAC CLI resolves the connection reference by filtering the Dataverse `connectionreferences` entity like this:

```
GET connectionreferences?$filter=connectionreferencelogicalname eq '<logicalName>'
                                 and solutionid eq '<solutionId>'
```

The `solutionid` **field on the `connectionreference` entity** stores the solution the record was *created in* (the Default Solution, `fd140aae-4df4-11dd-bd17-0019b9312238`), not the solution that **contains it as a component**. When you pass your custom solution's GUID as `-s`, the filter never matches — even if the connection reference is inside that solution.

**Resolution**

Pass the **Default Solution ID** (`fd140aae-4df4-11dd-bd17-0019b9312238`) as the `-s` value, regardless of which solution the connection reference belongs to:

```bash
pac code add-data-source \
  -a "shared_office365users" \
  -cr <connectionReferenceLogicalName> \
  -s fd140aae-4df4-11dd-bd17-0019b9312238
```

After the command succeeds, patch `power.config.json` so the `connectionReferences` key is the **logical name** of your connection reference (not a random GUID), keeping the config ALM-safe:

```json
"connectionReferences": {
  "cr289_officeconnref01": {
    "id": "/providers/Microsoft.PowerApps/apis/shared_office365users",
    ...
  }
}
```

> **Note:** The Default Solution ID (`fd140aae-4df4-11dd-bd17-0019b9312238`) is the same in every Power Platform environment — it is a well-known constant, not environment-specific.

---

### `pac code list-connection-references` crashes with `TypeError`

**Symptom**

```
TypeError: Cannot read properties of undefined (reading 'trackScenario')
```

**Resolution**

Disable PAC CLI telemetry before running the command:

```bash
pac telemetry disable
pac code list-connection-references -env <envUrl> -s <solutionId>
```

---

### `pac code list-connection-references` shows `Name: null`

**Symptom**

The **Logical Name** column prints `null` in PAC CLI v2.4.1 output even though the connection reference exists.

**Resolution**

Export the solution as a ZIP and read `customizations.xml`:

```bash
pac solution export --name <solutionUniqueName> --path ./export --overwrite
# Then open export/<solutionName>.zip → customizations.xml
# Search for <connectionreference connectionreferencelogicalname="..."
```

## Project structure

```
src/
  components/
    TeamDirectory.tsx   # Search UI + results grid
    TeamDirectory.css
    UserCard.tsx        # Individual person card with lazy photo loading
    UserCard.css
  hooks/
    useTeamDirectory.ts # Connector calls, debounced search, photo loading
  generated/            # ← created by pac code add-data-source (do not edit)
    services/
      Office365UsersService.ts
    models/
      Office365UsersModel.ts
  App.tsx
  main.tsx
```

## Key patterns used

| Pattern | Where |
|---|---|
| Connection reference (`-cr`) for ALM portability | `pac code add-data-source` step |
| Generated typed service — no `any` | `useTeamDirectory.ts` |
| Debounced search (400 ms) | `useTeamDirectory.ts` |
| Lazy photo loading per card | `UserCard.tsx` → `useEffect` + `onVisible` |
